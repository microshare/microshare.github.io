The Big Picture
What does the Bundling System do?
Turns IoT sensor readings into the right maintenance ticket for the right crew — automatically
Event-driven ticket lifecycle management via hierarchical config resolution and roster-based routing
📡
Sensor Fires
An IoT device (soap dispenser, trash sensor, motion detector) reaches a threshold and emits a data event.
⚙️
Rules Applied
The system fetches layered configuration — base defaults, solution rules, location overrides — and deep-merges them.
🎫
Ticket Managed
A work ticket is created, updated with new readings, or resolved. The right crew is automatically assigned.
Nashorn Bundler
Client-side JS on stream server. Orchestrates the full flow. Entry point: action(text, configObj)
Config Resource
Scala/Pekko HTTP. Exposes PATCH /config/{recType} for config evaluation.
Config Processor
Server-side Scala. Runs the 12-layer deep merge and matches roster records. processConfig()
Workflow Engine
Creates, updates, and resolves incident tickets. Receives roster assignments for crew notification.
Configuration
A hierarchy of rules
More specific settings always win — like dress-code rules cascading from company policy down to individual preference

Config Tree Structure

          

          

            
Deep Merge Semantics

            
Objects → Recursive merge
Keys from both sides preserved; nested objects recurse deeper into the tree

            
Arrays → Concatenated
Both sides appended — use this for task lists across config levels

            
Scalars → Later wins
More specific layer's value overrides the more general one

            
Location matching → Longest prefix
Best-matching device tag array wins. ["B","0","351","AS"] beats ["B"]

            
Timing fields → Always use PT format
Use PT168H not P1W — wrong format silently breaks ticket display

Each level is more specific than the last. You only set what you want to override — everything else inherits from above.

Key insight

              

                You only need to specify the fields you want to override. A location entry only
                needs priority: 20 — thresholds, timing,

Summary

        
Key Takeaways

        
The bundling system at a glance

      

      

        

          

            
🔀

            
12-Layer Config Merge

            
Default → User → Location. Each of 3 sources walks 4 specificity levels. More specific always wins. Only set what you need to override.

          

          

            
🎫

            
Smart Ticket Bundling

            
Multiple events at the same location join one open ticket. Control via avoidEvent, autoResolve, staffResolve, and deviceTag aggregation depth.

          

          

            
🗺️

            
Roster-Based Routing

            
Location trees + solution/alert/event filters route tickets to the right crew. Wildcards provide flexible coverage. 248 rosters at Zurich Airport.

          

        

        

          
Common Gotchas

          

            

              ⚠
              
Timing format: Always PT prefix (PT168H not P1W). Wrong format creates tickets that silently disappear from the UI.

            

            

              ⚠
              
Legacy field: Both incident and ticket are valid. configDefault() normalises incident→ticket but be consistent in new configs.

            

            

              ⚠
              
deviceTag aggregation: deviceTag: "3" groups events from all sub-rooms into one ticket. Intentional for large areas; accidentally set, it hides granularity.

            

            

              ℹ
              
Debug tip: Call PATCH /config directly to preview the full 12-layer merge and matched rosters without triggering a real ticket.
